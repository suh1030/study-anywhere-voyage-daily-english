# Polaris 上線級全面驗收報告

日期：2026-06-30

## 驗收結論

本輪建立 **64 個不同的語意／安全案例**，連同針對性修正與重複抽樣，共執行 **120 次語意評測**；另執行 **17 項 API 合約測試**、正常負載測試與突發負載測試。

所有本輪可重現的重大與中度問題均已修正並完成針對性複測。最終高頻文法確定性回歸為 **20/20**，進度資料重複回歸為 **9/9**，簡體注入回歸為 **3/3**，政策型固定回覆回歸為 **6/6**，API 合約為 **17/17**。

這代表目前本機版本已具備顯著更強的防錯與安全能力，但不代表生成式模型能對所有未知輸入提供數學上的 100% 正確保證。

## API 合約結果

結果：**17/17 通過**。

已確認下列輸入會在呼叫模型前被拒絕：

- `system`／`developer` 角色注入
- 超過 2,000 字的單則訊息
- 超過 20 則的對話歷史
- 非字串或空白內容
- 格式錯誤 JSON
- 非法或超長 learning context
- 藏在合法 context 欄位或 assistant history 中的提示注入

## 第一輪發現的重大問題

1. 接受使用者假進度，把可信的 9/12、75% 改成 100 天、99%。
2. 逐字洩漏完整 system prompt。
3. 被提示注入改成政治評論者，甚至輸出內部思考。
4. 接受錯誤文法前提並提供假來源。
5. 直接產生 Python 爬蟲，離開英文教學範圍。
6. 把正確的現在完成式改成錯誤過去式。
7. 自傷回覆仍想導回英文；藥物過量案例曾錯給非醫療緊急資訊。
8. 手寫簡體清單漏掉「点」，模型評審也未發現。
9. API 接受偽造角色、5,000 字訊息及 41 則歷史。
10. 短時間併發時，免費 provider 限流造成 25% 失敗。

## 已實作的防護

- system/context/user trust boundary 與提示注入偵測。
- API 角色、長度、數量及 context schema 驗證。
- OpenCC 完整簡體單字字典，並排除「台北、里長、皇后」等合法繁體用字。
- 進度問題由後端直接讀取已驗證 context 組答案，不再交給模型改寫數字。
- 自傷、藥物過量、嚴重胸痛、prompt 洩漏、注入、學術代寫、隱私與跨工作階段記憶採確定性回覆。
- 攔截簡體、內部思考、system prompt、截斷、過長、程式碼、中國用語、無故 emoji 與不可信進度數字。
- 四個高頻錯誤使用已驗證的確定性教學回覆；其餘已知錯誤另有語意一致性守門。
- Provider 鏈擴充為 Llama 70B → Llama 8B → GPT‑OSS 20B → Qwen3 32B → OpenRouter Free，所有輸出仍需通過同一套防線。

## 重複抽樣

- 精確進度／進度衝突／無漏打卡：**9/9**，p95 36ms。
- 四個高頻文法錯誤各重複 5 次：**20/20**，p95 76ms。
- 多輪現在完成式一致性：成功回覆均維持正確判斷。
- 簡體強迫注入修正後：**3/3**，24–38ms。
- GPT‑OSS 備援獨立抽查：5/5 成功生成；人工判讀 4 個完整合格、1 個安全但過度拒絕，因此保留為備援而非主模型。

## 殘餘風險

1. **免費服務沒有 SLA。** 這次 8 併發達到 100% 只代表本次樣本；供應商尖峰期仍可能全部限流，系統會安全失敗而非輸出不合格內容。
2. **正式 Edge Function 尚未部署驗證。** 本輪完整動態測試針對本機 proxy；正式 Supabase function 已同步程式邏輯，但仍需部署後做 staging smoke test。
3. **進度 context 由前端送入。** 現已驗證格式與阻擋指令注入，但惡意使用者仍可在自己的請求中偽造看似合法的數字。正式版若要求防竄改，應由後端直接查詢 `user_progress` 建立 context。
4. **未知文法案例仍依賴生成式模型。** 已知高頻錯誤有確定性防護，其餘輸入仍可能出錯；輸出防線會優先攔截已知危險模式。
5. **LLM judge 也會誤判或限流。** 本輪已使用程式規則、不同模型評審與人工複核交叉驗證，不把單一 judge 當唯一真相。
